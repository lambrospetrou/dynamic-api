# Plan: Vanity slugs + version-addressable URLs for `/apps/...`

Status: Phase 1 & Phase 2 implemented
Date: 2026-05-29
Scope: `src/index.ts`, `src/lib/*`, `src/do/AppDO.ts`, `public/ui/index.html`, `src/index.test.ts`

## Goal

Two related URL features for the execution plane (`/apps/...`):

1. **Vanity slug** — make app URLs human-recognizable by embedding a decorative
   slug next to the id. The id stays authoritative; the slug is cosmetic and
   ignored for routing. Lets an owner with many apps eyeball which app a URL
   belongs to (raw ids are unmemorable).

2. **Version addressing** — let a caller target a specific version or a named
   release channel (`production`, `active`, `latest`, or a pinned number)
   instead of always hitting the most-recent code.

Both live in the **first path segment** so the app's own path namespace
(everything after the first `/`) stays completely clean.

## URL grammar

```
/apps/<ref>/<app-path...>

<ref> := <appId>[ - <slug> ][ @ <selector> ]
```

- `<appId>` — the canonical app id. **Always exactly 32 lowercase hex chars,
  no hyphens** (generated by `crypto.randomUUID().replaceAll("-", "")`,
  `src/index.ts:41`). This invariant is what makes parsing unambiguous and must
  be preserved.
- `<slug>` — decorative only. Charset `[a-z0-9-]` (matches `CreateAppSchema`,
  `src/lib/schemas.ts:9`). **Ignored for routing.** Does not need to match the
  stored slug, does not need to be globally unique.
- `<selector>` — version selector after `@`. One of: a channel name
  (`production`, `active`, …), the reserved word `latest`, or a version number.
  Charset excludes `@`, so it is safe to split on.

### Why it parses unambiguously

- The id has **no `-`**, so the **first** `-` in `<ref>` is always the
  id/slug boundary.
- Neither id nor slug contains `@`, so the **last** `@` is always the
  ref/selector boundary.

### Worked examples

| URL                                         | appId     | slug          | selector     | resolvedSelector   |
| ------------------------------------------- | --------- | ------------- | ------------ | ------------------ |
| `/apps/3f2a…e1/forecast`                    | `3f2a…e1` | —             | —            | `active` (default) |
| `/apps/3f2a…e1-weather-bot/forecast`        | `3f2a…e1` | `weather-bot` | —            | `active`           |
| `/apps/3f2a…e1-weather-bot@latest/forecast` | `3f2a…e1` | `weather-bot` | `latest`     | `latest`           |
| `/apps/3f2a…e1-weather-bot@7/forecast`      | `3f2a…e1` | `weather-bot` | `7`          | `7`                |
| `/apps/3f2a…e1@production/forecast`         | `3f2a…e1` | —             | `production` | `production`       |

`selector` is the **raw** value as supplied after `@` (or `undefined` when the
user gave none). `resolvedSelector` is what the request actually resolves to
**after applying defaults** — i.e. `selector ?? DEFAULT_CHANNEL`. Keeping both
lets callers/logs see "what was asked" vs. "what was served".

## Parsing helpers (explicit functions)

New file `src/lib/ref.ts`. Pure, no I/O — trivially unit-testable.

```ts
export const APP_ID_RE = /^[0-9a-f]{32}$/;
export const DEFAULT_CHANNEL = "active"; // bare URL resolves here
export const LATEST = "latest"; // reserved virtual channel = MAX(version)

export type ParsedRef = {
    appId: string; // authoritative; everything downstream keys off this
    slug?: string; // decorative; undefined when absent
    selector?: string; // raw value after '@'; undefined when absent
    resolvedSelector: string; // selector ?? DEFAULT_CHANNEL
};

/**
 * Parse the first path segment of `/apps/<segment>/...` into its parts.
 * Returns null only when no plausible 32-hex appId is present at the front.
 *
 * Steps (order matters):
 *   1. Split off the version: take everything after the LAST '@' as `selector`.
 *   2. From the remaining ref, `appId` = chars before the FIRST '-'; the rest
 *      (after that '-') is the decorative `slug`.
 *   3. Validate appId against APP_ID_RE. Slug/selector are NOT validated here —
 *      slug is cosmetic, selector validity is decided at resolution time.
 */
export function parseRef(segment: string): ParsedRef | null;

/** Build the canonical vanity ref string for emitting URLs: `<id>-<slug>`
 *  (or just `<id>` when slug is empty/equals id). */
export function formatRef(appId: string, slug?: string): string;
```

Notes:

- `parseRef` returns `null` (→ caller responds 404) when the leading token is
  not a 32-hex id, so a typo'd/garbage ref fails fast and never spins up an
  empty Durable Object via `idFromName(garbage)`.
- Decode the segment once (it arrives via `c.req.param("id")`, already
  URL-decoded by Hono) before parsing.

## Phase 1 — Vanity slug (ship independently)

Smallest useful slice. No data-model changes; slug already exists on the record
(`AppRecord.slug`, registry `slug`). This phase ignores `selector` entirely
(treats every request as the current/latest version, preserving today's
behavior) — version handling lands in Phase 2.

### 1.1 Routing — `src/index.ts`

Both the auth middleware (`:247`) and the executor (`:266`) currently do
`const appId = c.req.param("id")` and use that raw segment for lookup, token
verification, and the LOADER cache key. Change to:

```ts
const parsed = parseRef(c.req.param("id"));
if (!parsed) return c.json({ error: "Not found" }, 404);
const { appId } = parsed; // authoritative id
```

Then:

- **Lookup**: `getAppRecord(c.env, appId)` — unchanged signature in Phase 1.
- **Token verify** (`:261`): pass `appId` (the extracted id), not the raw
  segment. This also fixes a latent bug — `verifyAppToken` compares the stored
  id to its `appId` arg (`src/lib/tokenCache.ts:23,28`), so passing a slug-y
  segment would have failed valid tokens.
- **Prefix stripping** (`:280-281`): keep using the **raw** matched segment, not
  the extracted id:

    ```ts
    const prefix = `/apps/${c.req.param("id")}`;
    url.pathname = url.pathname.slice(prefix.length) || "/";
    ```

    This already strips the entire first segment (id + slug + selector), leaving
    the correct app path. Do **not** rebuild the prefix from `appId` alone or the
    slice length will be wrong.

### 1.2 URL emission — `public/ui/index.html`

`:499` builds `const base = \`${location.origin}/apps/${app.id}/\``. Change to
use the vanity ref:

```js
const base = `${location.origin}/apps/${app.slug ? `${app.id}-${app.slug}` : app.id}/`;
```

(Keep the `?token=` test-URL logic at `:502` as-is.) Optionally surface the full
vanity URL in the app detail header (`:431`, `:721`).

### 1.3 Tests — `src/index.test.ts`

- Unit-test `parseRef`/`formatRef` directly (table from the examples above,
  plus: bare id, id+slug, id+slug@selector, id@selector, garbage → null,
  slug with internal hyphens like `weather-bot-v2`).
- Extend the exec-plane auth test (`:219+`) to hit
  `/apps/${appId}-some-slug/` and assert it routes/auths identically to
  `/apps/${appId}/` (same 401/200 outcomes).

### 1.4 Optional polish (defer unless wanted)

Canonical-slug redirect for **management/UI links only** (not exec requests —
avoid a redirect hop on every API call): if the slug-tail mismatches
`record.slug`, 301 to `formatRef(id, record.slug)`.

## Phase 2 — Version channels & selectors

Adds real meaning to `<selector>`. Builds on Phase 1's parser (`resolvedSelector`).

### 2.1 Data model — `src/do/AppDO.ts`

All versions are already retained in `app_versions` (+ `getVersion`,
`getHistory`). Add a channel-pointer table (new migration, `idMonotonicInc: 2`):

```sql
CREATE TABLE IF NOT EXISTS app_channels (
  name       TEXT PRIMARY KEY,   -- 'production', 'active', ...
  version    INTEGER NOT NULL,
  updated_at TEXT NOT NULL
);
```

- `latest` is **virtual** = `MAX(version)`; never stored.
- New RPCs:
    - `setChannel(name, version)` — promote/deploy a version to a channel.
    - `getChannel(name)` → version number or null.
    - `listChannels()` → for the UI.
- Replace `getCurrent()` usage with **`resolveVersion(resolvedSelector)`**:
    - numeric string → `getVersion(Number(sel))`
    - `latest` → version at `MAX(version)`
    - channel name → `getChannel(sel)` → `getVersion(that)`
    - **default (`active`)** → `getChannel("active")`, falling back to
      latest until the first explicit promote (smooth migration from today's
      always-latest behavior). Decide & document this fallback explicitly.
    - unknown channel / missing version → null → 404.

`getCurrent()` can remain as a thin alias for `resolveVersion(LATEST)` for the
management endpoints that legitimately want newest (create/update flows).

### 2.2 Lookup & caching — `src/lib/appCache.ts`

- Generalize `getAppRecord(env, appId)` →
  `getAppRecord(env, appId, resolvedSelector)`. The returned `AppRecord.current`
  becomes "the resolved version", not unconditionally newest.
- Cache key must include the selector: `${appId}@${resolvedSelector}`.
- TTL split: immutable numeric versions → long TTL; moving targets
  (`latest`, channels) → short TTL (current 60s is fine) so promotes/edits
  propagate. Keep `writeAppRecord` keyed by id for the write-through on
  create/update (those write the newest version).
- LOADER cache key (`src/index.ts:271`) → `${appId}_v${version}` (version is
  immutable, cleaner than `created_at`).

### 2.3 Management API — `src/index.ts`

- `PUT /api/apps/:id/channels/:name` `{ version }` → `setChannel` (promote/deploy).
- `GET /api/apps/:id/channels` → `listChannels` (UI).
- Executor passes `parsed.resolvedSelector` into `getAppRecord`.

### 2.4 UI — `public/ui/index.html`

- Channel selector / promote button on the app detail view.
- Quick-test URL builder can append `@<channel>` when a non-default channel is chosen.

### 2.5 Tests

- `resolveVersion` matrix: numeric / latest / channel / default-fallback / unknown.
- Exec routing for `@7`, `@latest`, `@active`, and bare (default).
- Promote endpoint moves what bare/`@active` serves.

## Phase 3 — Read consistency for the management plane

Workers KV GETs sit behind an edge read cache (>=60s default, floor ~30s even
with an explicit `cacheTtl`). Combined with the 60s in-memory layer, a `LATEST`
read can lag a write by up to ~the KV cacheTtl — the `put` in `writeAppRecord`
does not punch through a read cache already warm in another POP/isolate. Fine
for the execution plane (serving code is allowed to be eventually consistent),
but wrong for the management plane, where `PUT` regenerates a new version from
the *current* code as its base.

- **Authoritative reads** — `getAppRecord(env, appId, selector, { fresh })`
  bypasses memory + KV and reads the DO directly; `getAppRecordFresh(...)` is the
  convenience wrapper. Every `/api/apps/...` handler uses the fresh path; the
  execution-plane middleware/executor stay on the cached path. Fresh reads still
  repopulate the caches for later exec reads.
- **DO owns the patch** — codegen stays in the Worker (a multi-second
  `env.AI` call must not hold a Durable Object open and serialize all access to
  it). The Worker does a fresh read for the base code, generates, then hands the
  finished code to `AppDO.updateVersion(description, code, prompt, expectedVersion)`.
  The DO patches `app_meta.description` and inserts the next version with no
  intervening await (race-free numbering — it already did this), and the handler
  no longer passes `id`/`slug` the DO already holds.
- **Optimistic concurrency** — `expectedVersion` is the base the code was
  generated from; if a newer version landed meanwhile the DO returns
  `{ ok: false, reason: "conflict" }` and `PUT` responds `409`. `notfound` → 404.
  (Only meaningful now that the base read is authoritative.)
- `createVersion` remains the initial-create path (POST); `updateVersion` is the
  subsequent-version path (PUT).

## Invariants to guard

- **App id stays 32-hex, hyphen-free.** Parsing depends on it. If id generation
  ever changes, `parseRef` and `APP_ID_RE` must change in lockstep. Add a comment
  at `src/index.ts:41` pointing here.
- Slug remains **non-authoritative**: no global-uniqueness requirement, no
  slug→id index, no change to how the DO / KV / token map are keyed.

## Rollout

- Phase 1 is backward compatible: existing `/apps/<id>/...` URLs are unchanged
  (no `-`, no `@` → `{ appId, resolvedSelector: 'active' }`, and in Phase 1
  the selector is ignored so behavior is identical to today).
- Phase 2's only behavioral change is what a **bare** URL serves once a
  `active` channel is promoted. The latest-fallback keeps pre-promote
  behavior identical, so there is no hard cutover.
